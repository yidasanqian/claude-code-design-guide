# 제3장: 유닉스 철학과 CLI 전통

> "한 가지 일을 하되, 그것을 잘 하는 프로그램을 작성하라.
>  함께 동작하는 프로그램을 작성하라.
>  텍스트 스트림을 다루는 프로그램을 작성하라."
> —— Doug McIlroy, 유닉스 파이프 발명가

---

## 3.1 유닉스 철학: 50년의 설계 지혜

1969년, Ken Thompson과 Dennis Ritchie는 Bell Labs에서 유닉스를 만들었습니다. 이것은 단순한 운영체제가 아니라, 깊은 영향을 미친 **설계 철학**이었습니다.

유닉스 철학의 핵심은 세 가지 원칙으로 이루어집니다.

1. **단일 책임(Single Responsibility)**: 각 프로그램은 한 가지 일을 하되, 그것을 잘 합니다
2. **조합 가능성(Composability)**: 프로그램들은 표준 인터페이스(텍스트 스트림)를 통해 협력합니다
3. **투명성(Transparency)**: 프로그램의 동작은 예측 가능하고 관찰 가능합니다

이 세 가지 원칙은 50년이 지난 오늘날 Claude Code에서도 명확하게 나타납니다.

---

## 3.2 파이프: 가장 초기의 "도구 호출"

유닉스 파이프(`|`)는 인류 최초의 "도구 호출" 메커니즘 중 하나입니다.

```bash
# 가장 자주 사용한 명령어 10개 찾기
history | awk '{print $2}' | sort | uniq -c | sort -rn | head -10
```

이 명령어는 무엇을 할까요?

1. `history`가 명령어 히스토리를 출력합니다
2. `awk`가 두 번째 열(명령어 이름)을 추출합니다
3. `sort`가 정렬합니다
4. `uniq -c`가 중복을 계산합니다
5. `sort -rn`이 숫자 역순으로 정렬합니다
6. `head -10`이 처음 10줄을 가져옵니다

각 프로그램은 한 가지 일만 하고, 파이프를 통해 연결되어 복잡한 작업을 수행합니다.

**이것은 Claude Code의 도구 호출 체인과 놀랍도록 유사합니다.**

```
사용자 요청 → GrepTool(검색) → FileReadTool(읽기) → FileEditTool(수정) → BashTool(검증)
```

## 3.3 파일 시스템: 통합 추상화

유닉스의 또 다른 훌륭한 설계는 "모든 것은 파일이다"입니다. 네트워크 연결, 디바이스, 프로세스 정보—모두 파일 시스템 인터페이스를 통해 접근합니다.

이 추상화의 가치: **통합 인터페이스는 인지 부하를 줄입니다**.

Claude Code는 이 철학을 계승합니다. 도구 시스템(Tool System)도 통합 인터페이스입니다. 로컬 파일을 읽든, MCP(Model Context Protocol) 서버를 호출하든, 원격 작업을 실행하든, 모두 동일한 도구 호출 메커니즘을 통해 이루어집니다.

---

## 3.4 표준 입출력: 계약으로서의 인터페이스

유닉스 프로그램들은 stdin/stdout/stderr를 통해 통신합니다. 이 설계의 우아함은 다음에 있습니다.

- **분리(Decoupling)**: 프로그램은 데이터가 어디서 오고 어디로 가는지 알 필요가 없습니다
- **조합 가능성(Composability)**: 어떤 프로그램이든 다른 프로그램과 결합할 수 있습니다
- **테스트 가능성(Testability)**: 파일로 입력을 시뮬레이션하고, 출력을 캡처하여 결과를 검증합니다

Claude Code의 도구 시스템(Tool System)도 유사한 계약을 따릅니다.

```typescript
// 모든 도구는 통합 인터페이스를 가집니다
type Tool = {
  name: string
  description: string
  inputSchema: ToolInputJSONSchema  // 입력 계약
  execute(input: I, context: ToolUseContext): Promise<ToolResult<O>>  // 출력 계약
}
```

입력에는 스키마 제약이 있고, 출력에는 통합된 `ToolResult` 형식이 있습니다. 이것이 stdin/stdout의 현대적 버전입니다.

## 3.5 CLI의 황금기

1970년대부터 2000년대까지, CLI 도구들이 소프트웨어 세계를 지배했습니다. `vim`, `emacs`, `make`, `grep`, `sed`, `awk`—이 도구들은 설계가 충분히 훌륭하기 때문에 오늘날에도 여전히 사용됩니다.

CLI 도구의 장점:
- **스크립트 가능(Scriptable)**: 다른 프로그램에 의해 호출될 수 있습니다
- **조합 가능(Composable)**: 파이프와 리디렉션을 통해 결합합니다
- **낮은 오버헤드**: GUI 프레임워크가 필요 없습니다
- **원격 친화적**: SSH를 통해 어디서든 사용 가능합니다

Claude Code가 CLI를 주요 인터페이스로 선택한 것은 GUI가 나쁘기 때문이 아니라, 이러한 CLI 특성이 AI 에이전트(Agent)에게 매우 중요하기 때문입니다. **AI는 스크립트 가능하고, 다른 도구와 조합 가능하며, 서버에서 실행 가능해야 합니다**.

## 3.6 make: 가장 초기의 "태스크 오케스트레이션"

`make`는 1976년에 발명된 빌드 도구로, 핵심 개념은 **의존성 그래프**입니다.

```makefile
# 타겟: 의존성
app: main.o utils.o
    gcc -o app main.o utils.o

main.o: main.c
    gcc -c main.c

utils.o: utils.c
    gcc -c utils.c
```

`make`는 의존성을 분석하고 업데이트가 필요한 것만 다시 빌드합니다.

이 개념은 Claude Code의 태스크 시스템에 직접 반영됩니다. 태스크는 의존성, 상태(대기 중/실행 중/완료/실패), 병렬 실행 기능을 가집니다.

## 3.7 CLI에서 TUI로: 터미널의 진화

터미널 기능이 향상되면서 TUI(Terminal User Interface)가 등장했습니다. 이는 터미널에서 복잡한 인터페이스를 렌더링하는 기술입니다.

`vim`, `htop`, `ncurses` 애플리케이션들—이 도구들은 터미널이 텍스트를 표시하는 것 이상을 할 수 있으며, 풍부한 인터랙티브 인터페이스를 가질 수 있다는 것을 증명했습니다.

Claude Code는 **Ink**(CLI용 React)를 사용하여 TUI 인터페이스를 구축합니다. 이것은 중요한 기술적 선택입니다.

```tsx
// Claude Code의 UI 컴포넌트, React로 작성됨
function App() {
  return (
    <Box flexDirection="column">
      <Header />
      <MessageList messages={messages} />
      <InputBox onSubmit={handleSubmit} />
    </Box>
  )
}
```

React로 터미널 UI를 작성한다는 것은:
- 컴포넌트 기반, 재사용 가능
- 선언적 렌더링
- 상태(State) 기반
- 개발자에게 친숙한 프로그래밍 모델

을 의미합니다.

## 3.8 Claude Code에 구현된 유닉스 철학

| 유닉스 원칙 | Claude Code 구현 |
|------------|----------------|
| 단일 책임 | 각 도구는 한 가지 일만 합니다(`GrepTool`은 검색만, `FileEditTool`은 편집만) |
| 조합 가능성 | 도구들은 연결될 수 있으며, 출력이 다음 도구의 입력이 됩니다 |
| 투명성 | 모든 도구 호출이 사용자에게 표시되며, 검토하고 중단할 수 있습니다 |
| 텍스트 스트림 | 도구들은 구조화된 텍스트(JSON)를 통해 통신합니다 |
| 스크립트 가능 | 비인터랙티브 모드를 지원하며, 스크립트에서 호출될 수 있습니다 |
| 파이프 | stdin 입력을 지원합니다(`cat file | claude "이것을 분석해줘"`) |

---

## 3.9 요약

유닉스 철학은 세 가지 불변의 설계 원칙을 제시했습니다. 단일 책임, 조합 가능성, 투명성.

50년이 지난 지금도 이 원칙들은 좋은 도구를 만드는 기반입니다. Claude Code는 처음부터 새로 발명된 것이 아닙니다—유닉스 전통의 어깨 위에 서서, 이 원칙들을 AI 에이전트(Agent) 설계에 적용한 것입니다.

이 전통을 이해하면 Claude Code가 왜 GUI 대신 CLI를 선택했는지, 도구 시스템(Tool System)이 왜 이렇게 설계되었는지, 각 도구가 왜 명확한 단일 책임을 가지는지 이해할 수 있습니다.
