# 第 18 章：协调器模式

> 协调器是多代理系统的"指挥家"——它不演奏，但决定谁演奏什么。

---

## 18.1 什么是协调器模式

协调器模式（Coordinator Pattern）是多代理系统中的一种架构模式：

```
用户
  ↓
协调器（Coordinator）
  ├── 分析任务
  ├── 分解子任务
  ├── 分配给专业代理
  └── 汇总结果
        ↓
  专业代理 A（代码分析）
  专业代理 B（测试生成）
  专业代理 C（文档编写）
```

协调器本身不执行具体工作，而是负责**任务分解、代理调度和结果整合**。

## 18.2 Claude Code 的协调器实现

`src/coordinator/coordinatorMode.ts` 实现了协调器模式：

```typescript
// 协调器的用户上下文
export function getCoordinatorUserContext(
  mcpClients: ReadonlyArray<{ name: string }>,
  scratchpadDir?: string,

  return {
    coordinatorInstructions: buildCoordinatorInstructions(mcpClients, scratchpadDir),
  }

```

协调器模式通过特殊的系统提示激活，告诉 Claude 它的角色是协调者而不是执行者。

## 18.3 协调器的核心职责

**任务分析**：理解用户的高层目标，识别可以并行的子任务。

```
用户：重构整个认证系统，包括前端和后端

协调器分析：
- 子任务 1：分析现有认证代码（可以立即开始）
- 子任务 2：设计新的认证接口（依赖子任务 1）
- 子任务 3：实现后端认证（依赖子任务 2）
- 子任务 4：实现前端认证（依赖子任务 2，可与子任务 3 并行）
- 子任务 5：编写测试（依赖子任务 3 和 4）
```

**代理调度**：根据任务特性选择合适的代理类型。

```
子任务 1 → Explore 代理（只读，安全）
子任务 2 → Plan 代理（只生成计划）
子任务 3 → general-purpose 代理（有写入权限）
子任务 4 → general-purpose 代理（并行执行）
子任务 5 → general-purpose 代理（依赖前面的结果）
```

**结果整合**：收集各代理的输出，整合成最终结果。

---

## 18.4 Scratchpad：协调器的工作区

协调器使用一个专用的 scratchpad 目录来存储中间结果：

```typescript
// 协调器的 scratchpad 目录
const scratchpadDir = getScratchpadDir()
// 通常是 ~/.claude/scratchpad/<session-id>/
```

Scratchpad 的用途：
- 存储子任务的分析结果
- 记录任务依赖关系
- 保存中间状态（防止协调器上下文超限时丢失进度）

---

## 18.5 协调器 vs 普通代理

| 维度 | 普通代理 | 协调器 |
|------|---------|--------|
| 主要工作 | 执行具体任务 | 分解和调度任务 |
| 工具使用 | 文件操作、Shell 等 | 主要是 AgentTool |
| 上下文内容 | 代码、文件内容 | 任务计划、代理状态 |
| 输出 | 代码变更、分析结果 | 整合后的最终报告 |
| 适用场景 | 具体的编码任务 | 复杂的多步骤项目 |

---

## 18.6 协调器的局限性

协调器模式不是银弹：

**协调开销**：协调器本身需要消耗 token 和时间。对于简单任务，协调开销可能大于收益。

**信息传递损耗**：子代理的结果需要通过文件或消息传递给协调器，可能丢失细节。

**错误传播**：如果一个子代理失败，协调器需要处理失败情况，这增加了复杂性。

**调试困难**：多层代理的调试比单代理复杂得多。

---

## 18.7 何时使用协调器模式

**适合使用**：
- 任务明确可以分解为独立子任务
- 子任务可以并行执行，节省时间
- 任务规模超出单个代理的上下文窗口
- 需要不同专业知识的子任务

**不适合使用**：
- 任务简单，一个代理就能完成
- 子任务高度耦合，无法独立执行
- 对实时性要求高（协调器增加延迟）

---

## 18.8 小结

协调器模式是 Claude Code 多代理架构的高级用法：

- **职责分离**：协调器负责调度，专业代理负责执行
- **Scratchpad**：协调器的工作区，存储中间状态
- **适用场景**：复杂的、可分解的、可并行的大型任务

协调器模式体现了一个重要的工程原则：**复杂系统的管理本身也需要专门的角色**。
