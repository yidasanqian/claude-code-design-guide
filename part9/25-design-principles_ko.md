# 제25장: Claude Code 디자인 원칙

> 좋은 설계는 우연의 산물이 아닙니다. 일련의 의식적인 결정들이 만들어낸 결과입니다.

---

## 25.1 소스 코드에서 디자인 원칙 도출하기

Claude Code의 소스 코드를 심층 분석하면, 시스템 전체에 스며든 여러 디자인 원칙(Design Principle)을 도출할 수 있습니다. 이 원칙들은 Anthropic이 공식적으로 명시한 것이 아니라 코드에서 추론한 것입니다. 코드는 가장 정직한 설계 문서이기 때문입니다.

---

## 25.2 원칙 1: 편의성보다 투명성

Claude Code는 여러 곳에서 편의성보다 투명성을 선택합니다.

**도구 호출 표시**: 모든 도구 호출은 도구 이름, 파라미터, 결과를 포함하여 사용자에게 표시됩니다. 사용자는 최종 결과만이 아니라 Claude가 무엇을 하고 있는지 볼 수 있습니다.

**권한 프롬프트**: 위험한 작업을 자동으로 실행하지 않고 먼저 사용자에게 확인을 요청합니다. 마찰이 생기지만 사용자 제어권을 보장합니다.

**오류 투명성**: 도구 실행에 실패하면 오류 메시지를 숨기거나 미화하지 않고 사용자에게 완전히 표시합니다.

**비용 표시**: `/cost` 명령은 상세한 토큰 사용량과 비용을 표시하여 사용자가 실제 소비량을 파악할 수 있게 합니다.

**설계 통찰**: AI 에이전트(Agent) 시스템에서 투명성은 사용자 신뢰를 구축하는 토대입니다. 사용자는 AI가 무엇을 하는지 알아야 안심하고 더 많은 것을 맡길 수 있습니다.

---

## 25.3 원칙 2: 보안이 기본, 편의성은 선택

Claude Code의 기본 설정은 가장 안전한 상태입니다.

- 기본적으로 위험한 작업은 확인을 요구합니다
- 기본적으로 현재 작업 디렉터리에만 접근할 수 있습니다
- 기본적으로 어떠한 권한 검사도 건너뛰지 않습니다

사용자는 설정을 통해 제한을 완화할 수 있지만, 반드시 명시적으로 선택해야 합니다. 이것이 "기본 보안(Secure by Default)" 원칙의 구현입니다.

소스 코드에 반영된 모습:

```typescript
// 위험 모드는 명시적 플래그가 필요합니다
if (allowDangerouslySkipPermissions) {
  // 주의: 함수 이름에 "Dangerously"가 포함되어 개발자에게 위험임을 상기시킵니다
  logWarning('Running with --dangerously-skip-permissions')
}
```

`allowDangerouslySkipPermissions` 함수 이름의 "Dangerously"는 자의적인 명명이 아닙니다. 이것은 설계 결정으로, 명명을 통해 사용자에게 이것이 위험한 작업임을 상기시킵니다.

**설계 통찰**: 보안 시스템 설계는 "안전한 것을 하기"가 "안전하지 않은 것을 하기"보다 쉬워야 합니다.

---

## 25.4 원칙 3: 단일 책임, 복잡한 작업은 조합으로

Claude Code의 도구 시스템(Tool System)은 단일 책임 원칙을 엄격하게 따릅니다.

- `FileReadTool`은 파일 읽기만 담당합니다
- `GrepTool`은 콘텐츠 검색만 담당합니다
- `FileEditTool`은 문자열 치환만 담당합니다
- `BashTool`은 명령어 실행만 담당합니다

복잡한 작업은 "크고 포괄적인" 도구를 만드는 것이 아니라 Claude의 추론 능력이 이 원자적 도구들을 조합함으로써 완수됩니다.

이 원칙의 장점:
- 각 도구가 단순하고, 테스트 가능하며, 신뢰할 수 있습니다
- 도구들은 어떤 방식으로도 조합할 수 있습니다
- 새 도구를 추가해도 기존 도구에 영향을 주지 않습니다
- 도구 동작이 예측 가능합니다

**설계 통찰**: 에이전트(Agent) 시스템에서 도구는 원자적이어야 하고, 조합 로직은 도구 수준이 아닌 AI 수준에 있어야 합니다.

---

## 25.5 원칙 4: 암묵적보다 명시적

Claude Code의 코드에는 "명시적" 설계가 많이 있습니다.

**명시적 컨텍스트 전달**: 도구는 전역 변수를 통해 암묵적으로 접근하는 것이 아니라 `ToolUseContext`를 통해 모든 의존성을 명시적으로 받습니다.

**명시적 상태 업데이트**: 상태는 객체 속성을 직접 수정하는 것이 아니라 함수형 업데이트를 통해 명시적으로 수정됩니다.

**명시적 오류 처리**: 오류는 조용히 무시되지 않고 반환값이나 예외를 통해 명시적으로 전달됩니다.

**명시적 기능 플래그(Feature Flag)**: 기능 스위치는 환경 변수로 암묵적으로 제어되는 것이 아니라 `feature('FLAG_NAME')`을 통해 명시적으로 확인됩니다.

소스 코드의 주석도 이 원칙을 반영합니다.

```typescript
// DO NOT ADD MORE STATE HERE - BE JUDICIOUS WITH GLOBAL STATE
// (여기에 상태를 더 추가하지 마세요 — 전역 상태는 신중하게 사용하세요)
```

이 주석은 미래 개발자에 대한 경고입니다. 전역 상태는 암묵적 의존성의 원천이므로 가능한 한 피해야 합니다.

**설계 통찰**: 명시적 코드는 이해하고, 테스트하고, 유지보수하기 더 쉽습니다. 암묵적인 "마법"은 단기적으로는 편리하지만 장기적으로는 부담이 됩니다.

---

## 25.6 원칙 5: 실패를 위한 설계

Claude Code는 여러 곳에서 실패 시나리오를 고려합니다.

**도구 실행 실패**: 도구가 실패하면 오류가 도구 결과로 Claude에게 반환되고, Claude는 오류를 바탕으로 전략을 조정할 수 있습니다.

**API 호출 실패**: 재시도 메커니즘이 있으며, 재시도 가능한 오류(네트워크 타임아웃)와 재시도 불가능한 오류(인증 실패)를 구분합니다.

**인터럽트 처리**: 사용자가 중단할 때 `yieldMissingToolResultBlocks()`가 미완료 도구 호출에 대한 오류 결과를 생성하여 메시지 목록의 무결성을 보장합니다.

**컨텍스트 오버플로우**: 컨텍스트가 제한을 초과하여 실패하는 것을 방지하는 자동 압축(auto-compact) 메커니즘이 있습니다.

**예산 초과**: 토큰 및 비용 예산이 있으며, 초과 시 충돌 없이 우아하게 중지합니다.

**설계 통찰**: 분산 시스템과 AI 에이전트(Agent) 시스템에서 실패는 예외가 아닌 일반적인 상황입니다. 시스템 설계는 모든 작업이 실패할 수 있다고 가정하고 그에 상응하는 복구 전략을 갖추어야 합니다.

---

## 25.7 원칙 6: 관찰 가능성은 일급 시민

Claude Code는 완전한 관찰 가능성(Observability) 인프라를 갖추고 있습니다.

**OpenTelemetry 통합** (`src/bootstrap/state.ts`):
```typescript
tracerProvider: BasicTracerProvider | null
meterProvider: MeterProvider | null
loggerProvider: LoggerProvider | null
```

**진단 로그**: `logForDiagnosticsNoPII()`가 주요 작업을 기록하며, 개인 식별 정보(PII)는 포함하지 않습니다.

**성능 추적**: 시작 시간, API 지연 시간, 도구 실행 시간을 모두 추적합니다.

**비용 추적**: `cost-tracker.ts`가 각 API 호출의 토큰 사용량과 비용을 추적합니다.

**설계 통찰**: 관찰 가능성은 사후에 추가하는 것이 아니라 설계 초기부터 내재되어야 합니다. 관찰 가능성 없이는 시스템 동작을 이해할 수 없고, 최적화와 디버깅도 불가능합니다.

---

## 25.8 원칙 7: 점진적 복잡성

Claude Code의 설계는 사용자가 단순하게 시작하여 점차 더 복잡한 기능을 사용할 수 있도록 합니다.

**초급**: 자연어를 직접 입력하면 Claude가 작업 완수를 도와줍니다.

**중급**: CLAUDE.md로 프로젝트 컨텍스트를 설정하고, 슬래시 명령으로 효율성을 높입니다.

**고급**: MCP(Model Context Protocol) 서버를 구성하고, 커스텀 스킬(Skills)을 만들고, 멀티 에이전트(Multi-Agent) 협업을 사용합니다.

**전문가**: 권한 모델(Permission Model)을 설정하고, 샌드박스 모드를 사용하며, CI/CD 파이프라인에 통합합니다.

각 단계는 완전히 사용 가능하며, 현재 단계를 사용하기 위해 다음 단계를 이해할 필요가 없습니다.

**설계 통찰**: 좋은 시스템은 초보자에게 친절하면서 전문가에게 강력해야 합니다. 점진적 복잡성은 시스템을 시작하기 쉬우면서도 깊이 있게 만들어 줍니다.

---

## 25.9 원칙 8: 코드가 문서다

Claude Code의 코드에는 의미 있는 이름과 주석이 많습니다.

```typescript
// 함수 이름이 위험성을 나타냅니다
allowDangerouslySkipPermissions

// 주석이 설계 결정을 설명합니다
// Stable project root - set once at startup (including by --worktree flag),
// never updated by mid-session EnterWorktreeTool.
// Use for project identity (history, skills, sessions) not file operations.

// 주석이 제약 조건을 설명합니다
// DO NOT ADD MORE STATE HERE - BE JUDICIOUS WITH GLOBAL STATE

// 주석이 복잡한 규칙을 설명합니다
/**
 * The rules of thinking are lengthy and fortuitous...
 * 1. A message that contains a thinking or redacted_thinking block must be...
 */
```

이 주석들은 "무엇을 했는지"(코드 자체가 설명함)가 아니라 "왜 이렇게 했는지"와 "어떤 제약이 있는지"를 설명합니다.

**설계 통찰**: 좋은 주석은 의도와 제약을 설명하며, 코드를 반복하지 않습니다.

---

## 25.10 이 원칙들의 공통 주제

여덟 가지 원칙을 되돌아보면 공통 주제가 있습니다. **역량과 제어의 균형**입니다.

Claude Code는 강력한 도구이지만, 그 설계는 항상 사용자 제어를 최우선에 둡니다.
- 투명성은 사용자가 무슨 일이 일어나고 있는지 알게 합니다
- 기본 보안은 사용자가 사고를 걱정하지 않게 합니다
- 명시적 설계는 사용자가 동작을 이해하고 예측할 수 있게 합니다
- 관찰 가능성은 사용자가 모니터링하고 디버깅할 수 있게 합니다

이 균형은 AI 에이전트(Agent) 시스템 설계의 핵심 과제이며, Claude Code의 가장 중요한 설계 성과입니다.

---

## 25.11 요약

Claude Code의 여덟 가지 핵심 디자인 원칙(Design Principle):

1. **편의성보다 투명성**: AI가 무엇을 하는지 사용자가 볼 수 있게 합니다
2. **보안이 기본, 편의성은 선택**: 기본 보안(Secure by Default)
3. **단일 책임, 복잡한 작업은 조합으로**: 원자적 도구 + AI 조합
4. **암묵적보다 명시적**: 명확한 의존성, 상태, 오류
5. **실패를 위한 설계**: 모든 작업은 실패할 수 있으며, 모두 복구 전략이 있습니다
6. **관찰 가능성은 일급 시민**: 내장된 모니터링과 추적
7. **점진적 복잡성**: 초보자에게 친절하고 전문가에게 강력합니다
8. **코드가 문서다**: 의미 있는 명명과 의도를 설명하는 주석

이 원칙들은 AI 에이전트(Agent) 시스템뿐만 아니라 모든 복잡한 엔지니어링 시스템에 적용됩니다.
